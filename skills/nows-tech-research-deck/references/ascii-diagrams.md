# ASCII Diagrams · 架构图规范

**核心约束**：用 ASCII/Unicode 画架构图，**禁用 Mermaid**。
原因：（1）用户明确偏好 ASCII UI；（2）ASCII 图在任何终端、Markdown viewer、HTML 里都一致；（3）版本可控 diff 友好。

---

## 字符集（统一用这些，别混用）

| 用途 | 字符 | 示例 |
|---|---|---|
| 方框边 | `┌ ┐ └ ┘ ─ │` | `┌────┐` |
| 粗框（强调核心模块） | `╔ ╗ ╚ ╝ ═ ║` | `╔════╗` |
| 箭头 | `→ ← ↑ ↓ ↔ ⇢ ⇠` | `A → B` |
| 分叉 | `├ ┤ ┬ ┴ ┼` | `A ─┬─ B` |
| 虚线（可选路径 / 异步） | `┄ ┆ ╌ ╎` | `A ┄→ B` |
| 层级分隔 | `═══════` | `═════ 数据层 ═════` |

**不要混**：一张图要么全用细框，要么全用粗框（核心层粗、外围层细）。不要 50% 细 + 50% 粗。

## 5 种架构图范式

技术调研里 90% 的架构图能用这 5 种搞定，挑最合适的一种。

### 范式 1 · 分层架构（Layered）

适用：有明显"上层依赖下层"关系的系统（编译器、存储引擎、OS）。

```
┌──────────────────────────────────────────┐
│         用户接口层  CLI / UI / API       │
└──────────────────────────────────────────┘
┌──────────────────────────────────────────┐
│         应用逻辑层  Parser / Planner     │
└──────────────────────────────────────────┘
┌──────────────────────────────────────────┐
│         执行层    Compiler / Runner      │
└──────────────────────────────────────────┘
┌──────────────────────────────────────────┐
│         存储层    Adapters / Warehouse   │
└──────────────────────────────────────────┘
```

### 范式 2 · 流水线（Pipeline）

适用：数据/请求从左到右流动（ETL 工具、编译器、编排器）。

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ 源代码  │──→│ 解析    │──→│ 编译    │──→│ 执行    │
│ models/ │   │ Parser  │   │ Compiler│   │ Runner  │
└─────────┘   └─────────┘   └─────────┘   └─────────┘
                   │             │
                   ↓             ↓
              manifest.json  compiled SQL
```

### 范式 3 · 中心辐射（Hub & Spoke）

适用：有一个核心协调者 + 多个插件/适配器（dbt、Terraform、VSCode）。

```
                 ┌───────────┐
                 │  Snowflake│
                 └─────▲─────┘
                       │
        ┌──────────┐   │   ┌──────────┐
        │ Postgres │◄──┤──►│ BigQuery │
        └──────────┘   │   └──────────┘
                       │
                 ╔═════╧═════╗
                 ║  dbt-core ║
                 ╚═════╤═════╝
                       │
        ┌──────────┐   │   ┌──────────┐
        │ Redshift │◄──┴──►│ DuckDB   │
        └──────────┘       └──────────┘
```

### 范式 4 · 生命周期（Lifecycle / State Machine）

适用：讲"一次请求/任务的生命周期"。

```
 [提交] ──→ [解析] ──→ [编译] ──→ [执行] ──→ [完成]
   │           │          │         │          │
   └── fail ──┴── fail ──┴── fail ─┴── retry ──┘
                                       │
                                   [失败归档]
```

### 范式 5 · 双轴对比（Two-axis Map）

适用：把产品放到行业坐标系里（Act 3 VS 那一幕特别适合）。

```
                     ↑ 代码化
                     │
             dbt  ●  │  ● SQLMesh
                     │
                     │
  ───────────────────┼───────────────────→
     手工            │              编排强
                     │
          Airflow ●  │  ● Dagster
                     │
                     ↓ 配置化
```

## 尺寸与可读性

- **总宽 ≤ 80 字符**（屏幕和 HTML 都能不换行）
- **方框最小 8 字符宽**（太窄看不清内容）
- **同层模块宽度统一**（视觉对齐）
- **文字不要超出框**（溢出就换行或缩写）

---

## 常见翻车点

1. ❌ 用 ASCII 画复杂拓扑（>10 个节点）—— 换成范式 3 或分成多张
2. ❌ 箭头方向不统一 —— 同一张图要么都横着要么都竖着
3. ❌ 用中英文混框宽度 —— 中文字符占 2 格宽，画框时要数清楚
4. ❌ 试图用 ASCII 表达时间/并发 —— 那是 sequence 图该做的事，老老实实用范式 4 的状态机变体
