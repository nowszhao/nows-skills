# Tech101 — 技术傻瓜教程生成

将一个技术名词转化为一份让读者"看完立马就懂、立马能上手"的 101 级教程。最终产出一份排版清晰的 Markdown 稿件，用户需要时可额外生成 PDF。

## 核心交付标准

一份合格的 Tech101 教程必须让读者在读完之后能回答三个问题：

1. **这东西到底是什么，解决了什么之前解决不了或不好解决的问题？**（核心概念）
2. **它是怎么拼起来、怎么跑起来的？**（技术架构 + 技术原理）
3. **我现在就想试，从哪行代码开始抄？**（使用示例）

如果读者读完仍然要去 Google 一个更基础的概念，或者无法复制粘贴跑出 Hello World，教程就是不合格的。

## 前置准备

### 环境准备

1. **PDF 转换脚本**：本 Skill 自带 `scripts/md_to_pdf.py`（基于 WeasyPrint），用于将最终 Markdown 教程转为 PDF。仅当用户明确要 PDF 时才需要。依赖：`pip install weasyprint markdown --break-system-packages`。
2. **写作风格**：本 Skill 已内置完整写作风格指南（见下文"写作风格"部分），无需额外加载其他 skill。
3. **联网能力**：本 Skill 强依赖联网获取一手资料。如果环境中安装了 `web-access` skill，必须在调用子 Agent 时要求加载它；如果没有，使用 WebSearch/WebFetch 作为兜底。
4. **架构图方案**：使用 **ASCII UI 风格架构图**，零外部依赖，纯文本形式嵌入 Markdown 代码块，MD 和 PDF 都能直接渲染。**禁止使用 Mermaid**（视觉质量不足）。具体规范见下文"技术架构"章节。

### 明确研究对象

拿到用户输入后，先确认以下信息。如果用户给得足够明确（"帮我做一份 gRPC 的 101 教程"），不需要追问，直接开始：

1. **技术名词**：具体是什么——框架（React）、库（Pandas）、协议（WebSocket）、范式（Declarative ETL）、架构（Lakehouse）、工具（Dagster）、语言特性（Go generics）还是标准（OpenAPI）？
2. **读者假设**（可选）：读者是完全 0 基础的小白，还是有相关背景的工程师？默认假设读者是**有基础编程经验但对这个具体技术一无所知**的工程师。
3. **特别关注点**（可选）：有没有特别想看到的场景或对比？

## 第一步：联网信息收集

教程的质量取决于信息源的一手性和丰富度。**必须联网搜索**，不能只靠模型已有知识——很多技术的最新版本、最佳实践、API 签名在训练截止后已经变化。

### 信息源优先级（严格遵守）

优先英文资源，英文技术文档的质量和更新速度显著优于中文翻译：

| 优先级 | 类型 | 典型来源 |
|--------|------|---------|
| ★★★★★ | 官方文档首页 / Getting Started | 项目官网（如 react.dev、pytorch.org、grpc.io） |
| ★★★★★ | 官方 GitHub README + examples/ 目录 | github.com/org/repo |
| ★★★★ | 官方技术博客、Design Docs、RFC | project blog、Google/Meta/Apple engineering blogs |
| ★★★★ | 学术论文（技术有学术渊源时） | arXiv、USENIX、SIGMOD、OSDI 等会议原文 |
| ★★★ | 权威二手解读 | Martin Kleppmann、Jay Kreps、高质量个人技术博客、InfoQ 英文原创 |
| ★★★ | Stack Overflow 高票答案、GitHub Discussions | 用于补充常见坑和最佳实践 |
| ★★ | 中文技术博客 | 掘金、知乎高赞、CSDN 原创（注意时效性和准确性） |
| ★ | 聚合媒体、营销软文 | 通常跳过 |

**关键原则**：一个技术如果有官方 "Getting Started" 或 "Quickstart" 页面，**必须直接阅读该页面**，不能只靠搜索摘要。该页面通常就是官方认可的最佳 101 入门路径。

### 并行搜索策略

使用子 Agent 并行搜索来提高效率。建议分工：

- **子 Agent 1 — 官方一手资料**：官网 Getting Started、GitHub README、官方 Examples、最新稳定版本号、官方 Tutorial
- **子 Agent 2 — 原理与架构**：官方设计文档、架构图、底层论文、核心 Committer 或创始人的解读博客
- **子 Agent 3 — 社区与实战**：Stack Overflow 常见问题、真实 Hello World 示例、典型使用场景、与相近技术的对比（必要时才启用）

**子 Agent Prompt 要求**（写入每个子 Agent 的 prompt 中）：

> 本任务需要联网获取信息。**必须加载 web-access skill 并遵循其指引**（如果环境中已安装该 skill）。
>
> **工具选择**：
> - 发现阶段用 WebSearch 找到具体的官方文档 URL
> - 确定 URL 后用 WebFetch 或 Jina（r.jina.ai 前缀）提取正文
> - 需要原始 HTML（结构化元信息、JSON-LD）时用 curl
> - 官网文档有反爬、需登录、或静态抓取拿不到内容时，用 web-access 的浏览器 CDP 能力
>
> **信息源要求**：
> - 优先英文一手资料：官方文档 > GitHub README > 官方博客 > 权威二手解读
> - 中文资料仅作为补充参考，不作为主要依据
> - 学术类技术（算法、协议、分布式系统等）必须查 arXiv / 原始论文，调用方式：`curl -s "https://export.arxiv.org/api/query?search_query=all:关键词&max_results=10"`
> - 每个关键事实至少有一个可引用的 URL 来源
>
> **prompt 用词**：描述目标（"获取"、"调研"、"了解"），避免暗示具体手段（"搜索"、"抓取"），让子 Agent 自主判断最佳路径。

### 必须搞清楚的信息清单

搜索完成后自检。以下任何一项缺失，都要再补搜：

- [ ] 技术的**最新稳定版本号**和**发布日期**
- [ ] 官方的**一句话定义**（通常在官网 hero 区或 GitHub About）
- [ ] 官方的**Getting Started 路径**（从 0 到 Hello World 的步骤）
- [ ] 至少一个**最小可跑代码示例**（能复制粘贴直接运行的那种）
- [ ] **核心架构组件**及其关系（有图最好）
- [ ] **它解决的具体问题**（没有它时人们怎么做、痛点是什么）
- [ ] **与 2-3 个相近技术的定位差异**（读者脑中很可能已经有一个类似的东西）
- [ ] **典型使用场景**（至少 3 个真实场景）
- [ ] **常见坑和最佳实践**（来自 Stack Overflow 或 GitHub Issues 的高频问题）

信息不够就再补搜。宁可多搜一轮，也不能靠想象填空。

## 第二步：教程写作

教程的黄金结构是六段式。每一段都有清晰的任务，不可颠倒顺序。

### 1. 一句话定义 + 30 秒电梯版

开头第一段必须用**一句话**讲清楚这个技术是什么。这句话要做到：

- **不用行话解释行话**：如果定义里用了 B 概念来解释 A，那 B 必须是读者已经懂的东西
- **用类比降维**：把陌生技术映射到读者熟悉的东西上。"Dagster 之于数据工程，就像 React 之于前端——都是用声明式方式描述 `做什么`，让框架负责 `怎么做`。"
- **明确边界**：它**不是什么**往往比它是什么更重要。"Kafka 不是数据库，它是一条会记日记的消息河。"

一句话之后，再用 **150-300 字** 的"30 秒电梯版"展开：解决什么问题、核心思路是什么、现在谁在用。让读者即使只读这一段也能在会议室里装懂。

### 2. 它为什么会存在（背景与动机）

不讲背景的教程等于在空中盖楼。这一段要回答：

- **没有它之前，人们怎么干？** 具体到命令、代码、架构
- **旧方式的痛点是什么？** 不是抽象的"不够灵活"，而是具体的"写 500 行胶水代码还处理不好错误重试"
- **谁在什么场景下受不了了，于是搞了它？** 起源故事要简短但关键

这一段是教程的情绪钩子。读者如果认同痛点，后面所有内容都会读得更认真。控制在 300-500 字。

### 3. 核心概念（名词表）

列出使用这项技术必须先认识的 **5-10 个核心术语**。每个术语用以下格式：

```
**术语名**（英文原文）
一句话人话解释 → 可选的类比 → 它在整个体系里扮演什么角色
```

示例：

> **Asset**（资产）
> Dagster 里的 Asset 指一个"被产出的数据物件"——可以是一张表、一个文件、一个 ML 模型。你不是在 Dagster 里写 "先跑 A 再跑 B"，而是声明 "表 X 由表 Y 和 Z 产出"，框架自己推导执行顺序。把它想成 Makefile 里的 target，但长在数据工程里。

术语表是读者后面看架构图和代码示例时能对得上号的前提。宁可多列几个，也不要让读者半路被一个不认识的词卡住。

### 4. 技术架构（必须有图）

用 **ASCII UI 风格架构图** 画出核心组件关系，嵌入 Markdown 代码块（```text 或 ``` 围栏）。**禁止使用 Mermaid**——视觉质量不足；ASCII 图零外部依赖、纯文本、在任何终端 / MD 渲染器 / PDF 里都能显示。

#### ASCII 绘图规范（必须遵守）

**字符选择**：优先使用 **Unicode 盒形绘制字符**（U+2500 系列），比 ASCII `+-|` 更清晰：

- 直线：`─`（水平）、`│`（竖直）
- 拐角：`┌ ┐ └ ┘`（直角）、`╭ ╮ ╰ ╯`（圆角，视觉更柔和）
- 交叉 / T 型：`├ ┤ ┬ ┴ ┼`
- 粗线（强调主组件）：`━ ┃ ┏ ┓ ┗ ┛`
- 双线（强调边界 / 子系统）：`═ ║ ╔ ╗ ╚ ╝`
- 箭头：`→ ← ↑ ↓ ↔ ⇒`（数据流方向必须有箭头）

**对齐原则**：

1. **方框宽度统一**：同一层级的所有方框宽度一致，便于视觉对齐
2. **左右对齐**：通过空格填充让标签在方框内居中
3. **线条对齐**：方框之间的连线必须精确对接到方框边缘字符
4. **等宽字体渲染**：绘制时始终假设用户用等宽字体查看，每个中文字符占 2 列宽度，英文字符占 1 列

**信息密度**：

- 一张图 **最多放 7-9 个主要方框**，超过就拆分成多张图
- 每个方框里写 **组件名 + 一行简短说明**，不要挤太多内容
- 标签上方 / 下方可加一行小字说明数据流性质（"gRPC"、"async"、"write-ahead"）

#### 绘制流程

1. **选定模板**：根据技术类型挑上面三套之一做起点，不够用再自行组合
2. **先列组件清单**：在画图前把所有要出现的方框和连接列成文本清单，避免画到一半漏掉
3. **按模板画草稿**：用等宽字体在代码块里画出来
4. **对齐校验**：逐行用眼睛扫一遍，确认每根连线精确对到方框边缘，方框宽度一致
5. **配职责说明**：图下方用列表补充每个组件的职责、输入、输出

#### 常见错误（避坑）

- ❌ **混用字符集**：一张图里同时出现 `+-|` 和 `─│┌`，视觉极乱。选定 Unicode 盒形字符就一以贯之
- ❌ **连线不对齐**：箭头指向的位置不在方框边缘，看起来像线飘着。必须精确到字符列
- ❌ **方框宽度不一**：同层级的几个组件宽度不同，整张图歪歪扭扭
- ❌ **中英文混排忘了宽度差**：中文字符占 2 列，如果方框用中文标签，填充空格要按 2 列算
- ❌ **箭头方向不明**：没有箭头的连线读者看不出谁调谁。**所有连线必须有方向**，双向交互用 `◀──▶`
- ❌ **信息塞太多**：一张图超过 10 个方框就拆开。宁可画 2 张图，每张 5 个方框，也别塞一张大图

#### 图后必配：组件职责清单

ASCII 图只能表达拓扑结构，细节靠清单补。格式：

```markdown
**组件职责**：
- **Web UI**：dagster-webserver 进程，React 前端，通过 GraphQL 读取元数据、展示 Asset DAG
- **Metadata DB**：PostgreSQL，记录所有 Run、Asset Materialization、Event Log
- **User Code Server**：独立进程跑用户定义的 `@asset` 代码，通过 gRPC 暴露给 Webserver
- **Daemon**：常驻进程，负责 Schedule 触发、Sensor 轮询、Run Queue 调度

**数据流**：
1. 用户打开 Web UI → Webserver 通过 gRPC 向 User Code Server 拉取 Asset 定义
2. 用户点击"Materialize" → Webserver 写 Run 记录到 Metadata DB
3. Daemon 扫描 Run Queue → 调用 User Code Server 执行 → 结果写回 Metadata DB
```

如果技术本身是单体的（比如一个语言特性、一个算法），这一章可以改写成**"工作流程"**——从调用到返回，中间经历了什么。用模板 C（时序流程）最合适，左侧主体是调用方，右侧是被调方或各个内部模块。

### 5. 技术原理（为什么能 work）

这一章回答"凭什么它能做到这些"。根据技术类型选择不同切入点：

- **框架/库**：核心设计模式（如 React 的 Virtual DOM diff、Dagster 的 Asset-centric DAG）、关键数据结构、关键算法
- **协议/标准**：报文格式、握手流程、状态机
- **分布式系统**：一致性模型、选主机制、分区容错策略
- **算法/范式**：核心数学直觉、与朴素方法的对比、复杂度分析

**关键写作技巧：层层剥开**。先摆一个看起来反直觉的现象或特性，再追问"它是怎么做到的？"，一层一层往下挖。不要一上来就丢出完整原理——那会让读者被淹没。

控制在 800-1500 字。技术原理要讲透，但不要变成论文复述。读者的心理状态应该是"哦——原来是这么干的，牛"而不是"等等，再给我看一遍"。

### 6. 使用示例（必须能跑）

这是教程的最终价值兑现区。必须包含：

**6.1 环境准备**

具体到命令。版本号必须真实（来自搜索到的最新稳定版）。

```bash
# 示例：Dagster
pip install dagster dagster-webserver
```

**6.2 Hello World**

从官方 Quickstart 搬运或简化。**必须是一段可以复制粘贴就跑起来的代码**。强调：

- 文件名明确（`hello.py`）
- 运行命令明确（`dagster dev -f hello.py`）
- 预期输出明确（"浏览器打开 http://localhost:3000，你会看到一个带 1 个资产的 DAG"）

**6.3 一个真实一点的例子**

比 Hello World 再进一步一点，能体现这个技术的**核心卖点**。比如 Dagster 的 Asset 依赖传递、React 的受控组件、gRPC 的流式调用。控制在 30-60 行代码。代码之后要有**逐行或逐块讲解**，说清楚每一行在做什么、为什么这么写。

**6.4 常见坑 / 上手后的第一批问题**

列 3-5 个新手常踩的坑，每个坑一句话 + 解决办法。例如：

- "启动命令报 `module not found`" → 没激活 venv 或没 `pip install -e .`
- "`@asset` 装饰的函数不出现在 UI 上" → 忘了在 `Definitions` 里注册

### 7. 结尾：下一步去哪

2-3 段话，告诉读者：

- 官方文档的**进阶入口** URL（不是首页，是"Tutorial"、"Concepts Deep Dive"那一页）
- 值得订阅的**社区资源**（官方 Discord、Slack、GitHub Discussions）
- **下一个应该学的相关技术**（例如学完 Dagster 之后该看 dbt）

最后用**一句话扣回开头**（回环呼应）。让整篇教程有闭环感。

## 写作风格

这不是冷冰冰的咨询报告，而是一篇让人能从头读到尾的深度教程。写作风格需要在「教程的清晰」和「卡兹克的可读性」之间找到平衡点。

### 从卡兹克文风中借鉴的核心元素

**节奏感**：句子时长时短，段落之间跳跃自然。不要每段都一样长，一句话自成一段制造重量感的技巧可以用。好的节奏像波动，每次围绕主线偏出去一点，再用一句「扣主线句」拉回来。

**叙事驱动，不是罗列驱动**：介绍一个技术的诞生和原理时要有故事弧线，有起承转合。为什么是这一群人在这个时间点搞出了这个东西，背后的铺垫和触发点是什么。不要写成"2020 年发布了 1.0，2022 年发布了 2.0"这种流水账。

**知识是「聊着聊着顺手掏出来」的**：在讲述过程中自然地带出背景知识，不要「下面我来给大家科普一下」。当读者第一次遇到一个陌生概念时，在那一句话旁边用一个插入句自然解释掉，而不是开一个单独章节。

**敢下判断**：鼓励给出观点和洞察，但每个观点必须有事实支撑。先摆事实，再给判断。是推测的明确标注。表达判断时用「我觉得」「我的判断是」这种承认主观性的姿态，而不是居高临下的定论。

**层层剥开的修辞**：不直接讲结论，用"现象 → 表面解释 → 更深的追问 → 核心洞察"的方式展开。让读者参与到思考过程中。原理章节尤其要用这招。

**文化升维（克制使用）**：在核心洞察处偶尔连接到更大的文化/哲学/历史参照物。不是硬凑的升华，是「聊着聊着自然想到了」的感觉。教程类文章这招用 1-2 次就够，过多会喧宾夺主。

**回环呼应**：开头埋的类比或钩子，在结尾 callback 回来。前后因果的闭合感，是让教程从「信息流」变成「作品」的关键。

### 不从卡兹克文风中借鉴的元素

以下元素适合公众号文章但不适合技术教程，需要克制：

- **过强的口语化**：教程可以有聊天感，但不要满篇「这玩意」「不是哥们」「太牛逼了」。偶尔点缀可以，但密度要比公众号文章低很多。
- **去小标题化**：公众号文章追求一口气顺下来不加小标题。教程不一样，如果没有清晰的结构和导航，读者在找"环境准备在哪一段"时会崩溃。教程需要清晰的章节结构。
- **标点禁令可以放松**：公众号文章禁用冒号和破折号。教程中可以正常使用，因为教程需要的是信息传达效率。但「」的使用习惯可以保留。
- **固定尾部**：不要加公众号的三连/星标尾部。

### 绝对禁区

以下 AI 味标记无论什么文体都要避免：

- **套话**："首先...其次...最后"、"综上所述"、"值得注意的是"、"不难发现"
- **空洞形容词**："赋能"、"抓手"、"打造闭环"、"生态"滥用
- **教科书开头**："在当今 AI 快速发展的时代"、"随着技术的不断进步"
- **高频踩雷词**："说白了"、"意味着什么？"、"这意味着"、"本质上"、"换句话说"、"不可否认"
- **空泛工具名**：不说"某个框架"、"某种协议"，要说具体名字和版本号
- **编造场景**：如果某个信息搜不到，诚实标注「该信息暂缺，建议查阅 [官方文档 URL]」，绝不编造 API 签名、版本号、命令行

### 用人话写

避免技术文档式的套话和空洞概括。用具体的细节和例子代替概括性陈述。

- 不要写「该框架实现了高性能的数据处理」，要写「单机 8 核跑完 1TB Parquet 的 group by 需要 47 秒，比 Pandas 快 12 倍」
- 不要写「易于使用的 API」，要写「三行代码可以启动一个 REST server」
- 不要写「良好的社区支持」，要写「GitHub 38k stars，Discord 12k 人，核心 Committer 每周响应 Issue」

### 特别注意：代码示例的真实性

**所有代码示例必须是从官方文档、官方 GitHub、或经过验证可跑的真实代码**。不能编造 API 签名。如果在搜索过程中没有找到官方示例，必须在对应位置标注：

> **⚠️ 此处示例未经官方文档验证，仅根据 [来源 URL] 改写，使用前请对照 [最新官方文档 URL] 确认。**

- **版本信息**：本教程基于的技术版本号和发布日期

## 不同技术类型的适配

核心六段结构不变，但侧重点按类型调整：

**框架 / 库**（React、Dagster、Pandas）：重点在"使用示例"——占篇幅最大。原理章节聚焦设计模式（组件化、DAG、DataFrame）。

**协议 / 标准**（gRPC、WebSocket、OpenAPI）：重点在"技术架构"和"原理"——报文格式、握手流程、状态机必须有图。示例要包含完整的请求-响应一对。

**范式 / 方法论**（Declarative ETL、CQRS、Event Sourcing）：重点在"它为什么会存在"和"核心概念"——这类技术最容易被误解，必须把概念边界画清楚。代码示例要对比"旧范式" vs "新范式"的写法。

**工具 / 平台**（Dagster、dbt、Snowflake）：重点在"Hello World"——用户最想知道怎么跑起来看到第一个界面。架构图要画清楚各组件和用户的交互面（CLI、UI、API）。

**语言特性**（Go generics、Python async/await、Rust lifetime）：重点在"技术原理"和"常见坑"——这类技术的坑往往很深。示例要从"不用这个特性" → "用了这个特性" 的对比展开。

**算法 / 数据结构**（Raft、Paxos、Bloom Filter、LSM Tree）：重点在"技术原理"——需要讲清数学直觉和关键不变量。示例可以退化为伪代码，但必须有一个具体场景的完整演算（"假设有 5 个节点，过程如下..."）。

## 篇幅总览

| 部分 | 字数范围 | 说明 |
|------|---------|------|
| 一句话定义 + 30 秒版 | 200 - 400 字 | 必须精炼 |
| 它为什么会存在 | 300 - 500 字 | 情绪钩子 |
| 核心概念 | 500 - 1000 字 | 5-10 个术语 |
| 技术架构 | 500 - 1200 字 | 含图 + 组件清单 |
| 技术原理 | 800 - 1500 字 | 层层剥开 |
| 使用示例 | 1000 - 2500 字 | 含代码和逐行讲解 |
| 下一步去哪 | 200 - 400 字 | 带回环 |
| **全文总计** | **3500 - 7500 字** | 教程要完整但不啰嗦 |

如果某个技术本身很轻（比如一个单一 API），可以向下限靠拢；如果是复杂系统（比如 Kubernetes），可以向上限靠拢。超过 8000 字就要警惕——教程的目的是"快速上手"而不是"包打天下"。
